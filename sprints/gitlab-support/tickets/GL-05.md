# GL-05: Implement GitLab Client (Diffs, Comments, File Content)

## Description
Implement `GitLabPlatformClient` that fulfills the `PlatformClient` interface using GitLab's REST API v4. This is the core of GitLab integration — it handles fetching MR diffs, posting review comments, and reading file content.

## Affected Files
- **New**: `src/platforms/gitlab/client.ts` — GitLab API client implementation

## API Mappings

### `fetchDiff(owner, repo, prNumber)`
GitLab equivalent: Get the MR diff in unified diff format.
```
GET /projects/:id/merge_requests/:iid/diffs
```
- GitLab uses project ID (numeric) rather than `owner/repo`. Need to resolve `owner/repo` → project ID via `GET /projects/:owner%2F:repo`.
- Alternative: Use `GET /projects/:id/merge_requests/:iid/changes` which includes file diffs.
- The diff format may differ slightly from GitHub's unified diff — ensure `parseDiff()` in `src/review/differ.ts` handles it.

### `postReview(owner, repo, prNumber, commitSha, body, comments[])`
GitLab equivalent: Post MR discussions (inline comments).
```
POST /projects/:id/merge_requests/:iid/discussions
```
Body for inline comment:
```json
{
  "body": "comment text",
  "position": {
    "base_sha": "...",
    "head_sha": "...",
    "start_sha": "...",
    "position_type": "text",
    "new_path": "src/foo.ts",
    "new_line": 42
  }
}
```
For the summary comment (non-inline):
```
POST /projects/:id/merge_requests/:iid/notes
Body: { "body": "Review summary..." }
```

**Key difference**: GitHub batches all comments in a single `createReview` call. GitLab requires one API call per discussion/comment. This means N comments = N API calls. Consider rate limiting or batching.

### `fetchFileContent(owner, repo, path, ref)`
GitLab equivalent:
```
GET /projects/:id/repository/files/:path/raw?ref=:ref
```
- The file path must be URL-encoded (e.g., `src%2Ffoo.ts`)

## Implementation Notes
- Use `fetch` directly (no GitLab-specific SDK needed for REST calls)
- Authentication: `Authorization: Bearer <access_token>` header on all requests
- For self-hosted GitLab: base URL comes from `GITLAB_BASE_URL` config
- Project ID resolution (`owner/repo` → numeric ID) should be cached per-request to avoid redundant API calls
- Handle GitLab API rate limits (default: 300 requests/minute for authenticated users)

## Edge Cases
- GitLab MR diffs may use different path separators or include rename info differently
- Large MRs: GitLab paginates diffs (default 20 files per page) — handle pagination
- Binary files: GitLab marks files as `diff: "Binary files differ"` — ensure these are filtered

## Acceptance Criteria
- [ ] `fetchDiff` returns a unified diff string compatible with `parseDiff()`
- [ ] `postReview` creates inline discussions at the correct file/line positions
- [ ] `postReview` creates a summary note when review style includes summary
- [ ] `fetchFileContent` returns file content at a given ref
- [ ] All API calls include proper authentication headers
- [ ] Works with both gitlab.com and self-hosted GitLab instances
- [ ] Handles pagination for large MR diffs
