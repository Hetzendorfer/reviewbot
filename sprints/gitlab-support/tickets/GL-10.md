# GL-10: Testing — Integration Tests for GitLab Flow

## Description
Write tests covering the GitLab-specific implementation: webhook parsing, client methods, OAuth flow, and end-to-end pipeline execution with a GitLab platform context.

## Affected Files
- **New**: `tests/gitlab-webhook.test.ts` — webhook parsing and verification tests
- **New**: `tests/gitlab-client.test.ts` — client method tests (mocked HTTP)
- **New**: `tests/platform-registry.test.ts` — platform resolution tests
- **New**: `tests/pipeline-platform.test.ts` — pipeline with platform abstraction tests

## Test Cases

### Webhook Parsing (`gitlab-webhook.test.ts`)
1. **Valid MR open event** — parses correctly to `NormalizedPREvent`
2. **Valid MR update with commits** — triggers review (has `oldrev`)
3. **MR update without commits** — does NOT trigger review (label change, etc.)
4. **Draft MR** — filtered out, returns null
5. **Invalid token** — `verifySignature` returns false
6. **Non-MR event** — `parsePREvent` returns null
7. **Timing-safe comparison** — token verification uses constant-time comparison

### Client Methods (`gitlab-client.test.ts`)
Mock HTTP responses using Bun's built-in mocking or a test server.
1. **fetchDiff** — returns unified diff from GitLab MR diffs endpoint
2. **fetchDiff pagination** — handles multi-page diffs correctly
3. **postReview inline** — creates discussions with correct position payload
4. **postReview summary** — creates a note with review summary
5. **fetchFileContent** — returns decoded file content
6. **fetchFileContent 404** — returns null for missing files
7. **Self-hosted URL** — API calls use configured base URL

### Platform Registry (`platform-registry.test.ts`)
1. **Resolve github** — returns GitHub platform implementation
2. **Resolve gitlab** — returns GitLab platform implementation
3. **Unknown platform** — throws descriptive error

### Pipeline Integration (`pipeline-platform.test.ts`)
1. **GitHub job uses GitHub platform** — verify correct platform resolved
2. **GitLab job uses GitLab platform** — verify correct platform resolved
3. **Platform methods called in order** — createStatus → fetchDiff → LLM → postReview → markSuccess

## Implementation Notes
- Use `bun:test` framework consistent with existing tests
- Mock external HTTP calls — do not make real GitLab API calls in tests
- For pipeline tests, mock both the platform client and the LLM provider
- Use realistic GitLab webhook payloads (reference GitLab docs for exact shapes)

## Acceptance Criteria
- [ ] All test cases pass with `bun test`
- [ ] No real API calls made during tests
- [ ] Tests cover both happy paths and error cases
- [ ] Tests are deterministic (no flakiness)
- [ ] Coverage includes webhook verification, client methods, and pipeline integration
